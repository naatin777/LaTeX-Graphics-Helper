# AGENTS.md

このファイルは、Codex / AI coding agent に守ってほしいプロジェクト固有の作業ルールです。

## 基本方針

- このプロジェクトでは、完成させることを最優先する。
- 完璧な設計より、把握できる小さな変更を優先する。
- 1タスク1目的にする。
- feature implementation と refactoring を混ぜない。
- 大きな変更を一度に行わない。
- 既存の挙動を、明示的な指示なしに変更しない。
- dependency を追加しない。必要な場合は、理由と影響を先に説明する。
- ディレクトリ構成やアーキテクチャを、見た目の好みだけで変更しない。

## 作業前に読むもの

作業前に必ず以下を読む。

- `PROJECT_STATE.md`
- `docs/tasks/README.md`
- `docs/tasks/README.md` からリンクされた現在のタスク
- 関連する `docs/specs/`
- 必要なら `docs/adr/`

## AIへの制約

- `docs/tasks/README.md` からリンクされた現在のタスクだけを行う。
- 無関係なファイルを触らない。
- ついでの整理をしない。
- 仕様が曖昧な場合は、実装せずに確認・提案を行う。
- 変更範囲が大きくなりそうな場合は、実装前に分割案を出す。

## リファクタリング方針

リファクタリングは以下の条件を満たす場合のみ行う。

- バグリスクを下げる
- 次の機能追加が明らかに楽になる
- テストしやすくなる
- すでに理解しづらくなっている箇所を小さくする

以下の理由ではリファクタリングしない。

- なんとなく気持ち悪い
- 命名が好みではない
- もっと綺麗にできそう
- 軽微な重複がある
- MVP前に構成を整えたい

気になる点は、すぐ直さず `docs/refactor-backlog.md` に記録する。

## 重複の扱い

- 2回の重複は許容する。
- 3回目に同じ問題が出たら共通化を検討する。
- 仕様が揺れている間は、早すぎる共通化を避ける。
- 同じに見えて意味が違うものは共通化しない。

## テスト方針

- 重要な仕様に対してテストを書く。
- 内部実装ではなく、外から見た振る舞いをテストする。
- UIの細部やまだ揺れている仕様のテストは後回しでよい。
- カバレッジを追う前に、主要な挙動を安定させる。

優先してテストするもの。

- データ変換
- path handling
- command behavior
- configuration behavior
- error handling
- LaTeX code generation
- PDF / image conversion logic

## コメント方針

- コメントアウトしたコードを残さない。
- コメントは「何をしているか」ではなく「なぜ必要か」を書く。
- TODO には、対応する条件を書く。

良い例:

```ts
// TODO: 基本の変換フローが安定したら、batch cancellation に対応する。
```

悪い例:

```ts
// TODO: あとで綺麗にする。
```

## README / 言語方針

- メンテナは日本語で仕様・判断を書く。
- `README.ja.md` を意味の正本として扱ってよい。
- `README.md` は `README.ja.md` を元に自然な英語へ翻訳する。
- 英語READMEでは、実装以上に機能を盛らない。
- 作業用ドキュメントは日本語でよい。

## 作業後の報告

変更後は以下を説明する。

- 何を変更したか
- なぜ変更したか
- どう確認するか
- 仕様変更があるか
- 次にやるべきこと
