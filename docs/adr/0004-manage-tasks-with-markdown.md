# ADR-0004: タスクをMarkdownファイルで管理する

## ステータス

採用

## 日付

2026-06-21

## 背景

タスクの目的、変更範囲、完了条件が明文化されていないと、作業中に目的が広がり、無関係な変更や追加作業が混ざりやすい。

一方、現時点ではGitHub Issuesなどの外部管理を導入すると、登録や同期の手間が増え、個人開発に対して管理が重くなる。

タスクの履歴をリポジトリ内に残し、Codexも同じ情報を参照できる、単純な管理方法が必要である。

## 決定

タスクは `docs/tasks/*.md` に、1タスクにつき1つのMarkdownファイルを作成して管理する。

各タスクには `Status`、目的、完了条件、変更可能なファイルを記載し、完了したら `Status` を `Done` に変更する。

現在のタスクは `docs/tasks/README.md` からリンクし、ルートに別のタスク管理ファイルは置かない。

## 理由

- タスクの目的と範囲を作業前に固定できる
- タスクの状態と履歴をGitで管理できる
- Codexとメンテナが同じタスク定義を参照できる
- 外部サービスを使わず、テキストエディタだけで更新できる
- 仕様や設計判断へのリンクをタスクと一緒に残せる

## 代替案

### `NEXT.md` だけで現在のタスクを管理する

メリット:

- 管理するファイルが1つで済む
- 現在の作業をすぐ確認できる

デメリット:

- 更新時に過去のタスク内容が失われる
- タスクごとの完了条件や関連資料を履歴として残しにくい

### GitHub Issuesで管理する

メリット:

- 状態、ラベル、担当者などを管理できる
- コメントや関連Pull Requestをまとめられる

デメリット:

- 現在の個人開発には管理機能が多すぎる
- リポジトリ内のファイルだけではタスクを確認できない
- タスク登録や状態同期の手間が増える

## 結果・影響

- タスクごとにMarkdownファイルを作成、更新する手間が増える
- 完了したタスクの目的、範囲、判断を履歴として確認できる
- タスク数が増えた場合、ファイル一覧を整理する方法が必要になる可能性がある
- 外部のIssue管理機能は利用しないため、検索や集計はファイルとGitに依存する

## 運用ルール

- `docs/tasks/` に1タスク1ファイルで作成する
- `docs/tasks/README.md` には現在のタスクを1つだけリンクする
- 各タスクに少なくとも `Status`、目的、完了条件、変更可能なファイルを書く
- 作業中のタスクは1つに限定する
- タスクの目的を作業途中で増やさない
- 仕様に関係する場合は、関連する `docs/specs/` をリンクする
- 設計判断が必要な場合は、関連する `docs/adr/` をリンクする
- 完了条件を満たしたら `Status` を `Done` にする
- 残った作業は現在のタスクへ追加せず、別のタスクとして作成する

## 見直す条件

- タスク数が増え、Markdownファイルだけでは検索や優先順位付けが難しくなったとき
- 複数人で担当や進捗を共有する必要が生じたとき
- GitHub Issuesや他の管理ツールとの二重管理が発生したとき
