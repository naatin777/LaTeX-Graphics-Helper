# ADRの運用方針

このdirectoryには、このprojectで採用した重要な設計判断と、その判断理由を記録する。

ADRの目的は、現在の仕様や作業手順をすべて集約することではない。後から「なぜこの選択をしたのか」「どのtrade-offを受け入れたのか」を確認できるようにすることである。

## ADRに記録するもの

ADRは、原則として1fileにつき1つの永続的な判断を扱う。

次の条件に該当する判断を記録する。

- 複数の実装やtaskへ継続的に影響する
- 成立する選択肢が複数あり、trade-offがある
- 後から理由を確認する可能性が高い
- architecture、security、compatibility、data、開発運用の境界を決める
- 判断を変更したときに、影響範囲を追跡する必要がある

ADRには、判断を理解するために必要な次の情報を残す。

- 背景と解く問題
- 採用した判断
- 判断理由
- 実際に検討した代替案
- 受け入れる結果・影響
- 見直す条件
- 関連するADR、spec、task

見出しを埋めるためだけに、検討していない代替案や推測を追加しない。

## ADRに記録しないもの

| 情報                                 | 正本                                           |
| ------------------------------------ | ---------------------------------------------- |
| 利用者から見た機能仕様               | `docs/specs/`                                  |
| 現在行う変更、変更可能file、完了条件 | `docs/tasks/`                                  |
| 外部仕様やdependencyの調査結果       | `docs/research/`                               |
| AIが常に守る短いproject rule         | `AGENTS.md`                                    |
| 反復する操作手順                     | skillまたは専用guide                           |
| test fixture、runner、確認観点の詳細 | `docs/specs/internal/test-policy.md`またはtask |
| 一時的なbranch、PR、導入順、roadmap  | taskまたはproject rule                         |
| 個人のmodel、承認頻度、認証情報      | 個人設定                                       |

実装手順、command一覧、チェックリスト、設定項目一覧を、設計判断の代わりにADRへ置かない。これらが判断理由の理解に必要な場合は要点だけを書き、詳細の正本へlinkする。

## ADRを作らない場合

次の内容は、通常はtask、commit、code、testで十分である。

- 選択肢のない小さなbug fix
- 1箇所だけに閉じた実装詳細
- 一時的な調査手順
- まだ採用判断をしていない案
- 好みだけを根拠にしたformat、命名、directory整理

判断が必要か不明な場合は、先にtaskへ問題と選択肢を書く。採用する永続判断が決まった時点でADRを作る。

## 状態

- `提案中`: 判断を検討している
- `採用`: 現在有効な判断
- `却下`: 検討したが採用しなかった
- `置き換え済み`: 新しいADRによって判断が置き換えられた

判断を変更するときは、原則として既存ADRを現在の考えへ書き換えない。新しいADRを作り、旧ADRを`置き換え済み`にして相互にlinkする。

誤字、壊れたlink、事実誤認、判断を変えない短い補足は既存ADRで修正してよい。履歴としての意味が変わる編集は行わない。

## 分割の判断

行数だけでは分割しない。次の状態になった場合に分割または別文書への移動を検討する。

- 独立して変更され得る複数の決定がある
- 一時的な運用と永続判断が混在している
- 手順や一覧が判断理由より長くなっている
- 1つの決定を置き換えるために、無関係な判断まで更新する必要がある
- 正本がADRとspec、task、ruleの複数箇所に重複している

## 作成手順

1. `0000-template.md`を新しい番号と内容が分かる英語file名でcopyする。
2. 1つの判断だけを書く。
3. 関連するtask、spec、既存ADRへlinkする。
4. 採用後も残す必要のない手順やroadmapが混ざっていないか確認する。
5. 判断を実装するtaskとは別に管理してもよいが、ADR自体を未完了作業の代用にしない。
