## Canonical vocabulary

| Meaning                                | Canonical token        | Existing public/legacy token                       | Decision                                                                                                                                  |
| -------------------------------------- | ---------------------- | -------------------------------------------------- | ----------------------------------------------------------------------------------------------------------------------------------------- |
| independent input-to-output conversion | `convert`              | `convertTo*`, legacy `convertPngToPdf`             | Use `convertToPdf` for the generic PDF operation. Keep the old internal command ID and fallback key.                                      |
| image inputs to one PDF                | `combine`              | `convertImagesToSinglePdf`                         | Use `combineImagesToPdf` for internal symbol/file names. Keep the public command ID and setting key until a public migration is approved. |
| PDF inputs to one PDF                  | `merge`                | `mergePdf`                                         | Keep `mergePdf`; it already matches the input and result.                                                                                 |
| one PDF to many PDFs                   | `split`                | `splitPdf.allPages`, `splitPdf.configure`          | Use `splitPdfAllPages` and `splitPdfByPageGroups` for internal operations.                                                                |
| native Draw.io outputs                 | page PDFs / single PDF | `convertDrawioToPdf`, `convertDrawioToPdfDirectly` | Use result-based internal entrypoint names. Keep both public IDs and the `outputPath` keys.                                               |
| conversion run temporary area          | `staging area`         | `artifact`, `staging`                              | `staging` is the lifecycle term; keep path labels unchanged for safety.                                                                   |
| completed output before commit         | `prepared output`      | `staged output`                                    | Keep `PreparedConversionOutput`; distinguish it from staged input and intermediate files.                                                 |
| final user path                        | `final output`         | `output`                                           | Use in docs and audit; existing `outputPath` identifier remains public.                                                                   |
| existing output saved before overwrite | `backup`               | `previous file`                                    | Keep `previousFilePath` in code for compatibility with the current undo record; document it as a backup.                                  |

## Directory decision

### Before

```text
src/
  application/
  commands/
  config/
  edit_provider/
  operations/
  presentation/
  security/
webview/apps/{crop_pdf,merge_pdf,split_pdf}/
docs/{specs,adr,tasks,research,records,foundation}/
```

### After this phase

```text
src/
  application/
  commands/              # command entrypoints; split_pdf_commands.ts for split modes
  config/                # configuration readers, including all external tool paths
  edit_provider/
  operations/            # format operations and shared conversion lifecycle
  presentation/
  security/
webview/apps/{crop_pdf,merge_pdf,split_pdf}/
docs/{specs,adr,tasks,research,records,foundation}/
```

The proposed `conversion/`, `pdf/`, `external_tools/`, and `protocols/` tree was considered but not adopted. Current Oxlint boundaries already distinguish extension commands, core operations, configuration, security, and Webview apps; moving directories now would increase import and review cost without making a new dependency direction verifiable. `application/` remains the small home for source-format policy, protocol definitions, and Safe Mode state; it does not own VS Code command registration or conversion file writes. `operations/` owns format-specific operations and the shared staging/commit lifecycle; it does not own command registration or Webview rendering.

## Deferred items

- `LineOutputChannel` should move out of `external_tool_ascii_scratch.ts` only when the output channel contract changes or another generic type is added. The current fan-out would create a broad review-only diff.
- `ConversionArtifactRoot` should become `ConversionStagingArea` when the cleanup API is next changed. Renaming it alone would touch safety-sensitive lifecycle tests without changing behavior.
- Public command IDs, setting keys, pair-specific output templates, NLS keys, and historical task wording are not bulk-renamed in this phase.
