# Browser Playwright / VS Code Electron case overlap

- 対象: `test/playwright/webview-pdf-rendering.spec.ts` (Browser, 18 statically declared cases) と `test/playwright/electron/crop_pdf_configure.spec.ts` (Electron, 1 statically declared case)
- 状態: 監査用baseline。Browser削除、Electron全面移行、正本runnerは決めない
- 判定軸: oracle、runtime、failure modeが同じか。画面上の操作が似ているだけでは重複としない

## Harness facts

Browser specは`media/webview/crop_pdf/index.html`をローカルNode HTTP serverからChromiumへ配信する。PDFはテスト内で`pdf-lib`から生成するか、固定crop fixtureを読み込む。VS Code Hostは起動せず、`MessageEvent`と`acquireVsCodeApi().postMessage`をtest内で擬似化する。PDF.js worker、canvas、DPI、IntersectionObserver、layout、scroll、zoomを実Browserで確認するが、実VS Code URI、CSP enforcement、Host message bridgeは証明しない。

Oracle分類はrunner名と分ける。Vitestはrunner候補、Nodeはruntime、jsdom/happy-domはDOM environment候補、Playwright Browserはreal browser runner、Electronはactual VS Code runtimeである。

Electron specはVS Code `1.128.0`をdownloadしてElectronで起動する。development extension pathまたは`LGH_VSIX_PATH`のinstalled VSIXを使い、Explorer context menuから実commandを開く。実Webview frame、VS Code theme class/CSS variables、canvas、Apply後のPDF、notificationを確認する。packaged modeでは同じcaseの後半で、controlled external fetchが失敗すること、packaged internal module import、merge、Sharp PNG→JPEG、missing `pdftocairo` errorも確認する。これは全network requestが存在しないことや完全offline動作全体を証明するものではない。

Browser specはscreenshotを使わない。Electron specはLinux dark/light snapshotを2枚使い、failure時はwindow screenshot/console/frame textをartifactへ添付する。Browserのlocal HTTP serverは外部HTTPサービスではない。
